🤔 Primero: ¿Por qué necesitamos crear un "proyecto"?

Cuando trabajas con HTML, CSS y JS, normalmente creas archivos y los abres directamente en el navegador. ¿Por qué React es diferente?

  • React usa JSX: Es HTML dentro de JavaScript, pero los navegadores no lo entienden directamente.
  • Necesita traducción: Un "traductor" convierte tu código JSX a JavaScript que el navegador sí entiende.
  • Maneja dependencias: React necesita librerías adicionales que deben organizarse correctamente.

Piénsalo como cocinar: necesitas una cocina equipada (el proyecto) antes de empezar a cocinar (programar).

🔧 Paso 1 · Preparando nuestra "cocina"

Vamos a usar Vite para crear nuestro proyecto. Vite es como un chef experimentado que prepara toda la cocina por nosotros.

¿Por qué usamos Vite? Vite es una herramienta moderna que nos permite crear proyectos de React de forma rápida y sencilla. Es mucho más rápido que otras opciones como Create React App, porque:

  • 🚀 Arranca el proyecto en segundos (no tienes que esperar mucho para empezar a programar).
  • 🔄 Actualiza el navegador automáticamente cada vez que guardas cambios.
  • 📦 Organiza todas las dependencias y configuraciones por ti.
  • 🧑‍🍳 Es fácil de usar y recomendado por la comunidad de React.

En resumen, Vite nos ahorra tiempo y nos permite centrarnos en aprender React sin preocuparnos por configuraciones complicadas.

🏗️ Comando 1: Crear la estructura

npm create vite@latest mi-app -- --template react

¿Qué hace esto?

  • npm create vite@latest → Le dice a npm "usa la herramienta Vite más reciente"
  • mi-app → El nombre de tu proyecto (puedes cambiarlo por el que quieras)
  • --template react → "Prepárame todo lo necesario para React"
  • --/code Cuando te pregunte si instala algo experimental le dices que no

Después de ejecutar este comando, verás que se crea una carpeta llamada mi-app (o el nombre que hayas elegido) con varios archivos dentro.

Ya nos ha creado la estructura, ahora veremos para que vale cada carpeta y cuales podemos eliminar.

📁 Comando 2: Entrar al proyecto

cd mi-app

¿Qué hace esto? cd significa "change directory". Es como entrar físicamente a la carpeta de tu proyecto.

📦 Comando 3: Instalar las herramientas (se ha hecho automaticamente al crear la app)

npm install

¿Qué hace esto? Descarga todas las "herramientas" que React necesita:

  • El traductor de JSX a JavaScript
  • El servidor de desarrollo que actualiza automáticamente
  • React mismo y sus dependencias

🤔 Pregunta frecuente: "¿Tengo que instalar las dependencias cada vez?"

Respuesta corta: ¡NO! Solo necesitas hacer npm install en estas situaciones:

  • ✅ La primera vez que creas el proyecto
  • ✅ Cuando añadas nuevas librerías (como Material-UI, React Router, etc.)
  • ✅ Si borras accidentalmente la carpeta node_modules
  • ✅ Si descargas un proyecto de GitHub u otro lugar

❌ NO necesitas hacer npm install cada vez que:

  • ❌ Abres Visual Studio Code
  • ❌ Enciendes tu computadora
  • ❌ Modificas archivos como App.jsx
  • ❌ Trabajas en tu proyecto día a día

💡 Piénsalo así: Es como instalar un programa en tu computadora. Lo instalas una vez y ya está disponible para usar siempre.

🚀 Comando 4: Encender la cocina

npm run dev este será el comando que uses cuando entres y salgas de tu app.

¿Qué hace esto? Enciende un servidor local que:

  • Traduce tu código JSX automáticamente
  • Actualiza el navegador cuando guardas cambios
  • Te muestra errores si algo está mal

🎯 Resultado: Verás una dirección como http://localhost:5173. ¡Ábrela en tu navegador!

🔄 Tu flujo de trabajo diario (después de la primera instalación):

  1. Abres tu terminal en la carpeta de tu proyecto
  2. Ejecutas: npm run dev (¡solo este comando!)
  3. Abres tu navegador en http://localhost:5173 (si estas ocupado usará otro)
  4. Programas y guardas cambios → ¡Se actualizan automáticamente!
  5. Cuando termines: Ctrl+C en el terminal para parar el servidor

📝 Resumen: Después del primer día, solo necesitas npm run dev para empezar a trabajar.

Comando 5: npm install react-router-dom

Vamos a instalar React Router para manejar las rutas en nuestra aplicación. Ejecuta el siguiente comando en tu terminal:

npm install react-router-dom

¿Qué hace esto? Añade React Router a tu proyecto, permitiéndote crear múltiples páginas y navegar entre ellas sin recargar la página. Igualmente te permite llamar a tus componentes y renderizarlos en tu página. Una vez instalado vuelve a arrancar el servidor con npm run dev.

📂 Paso 2 · ¿Qué significa cada carpeta y archivo?

Cuando creas tu proyecto con Vite y React, se generan varias carpetas y archivos. Entender su propósito te ayudará a trabajar mejor:

🗂️ Carpetas principales

  • src/ — El corazón de tu app
    Aquí va todo tu código fuente: componentes, páginas, estilos personalizados, imágenes, etc. Es la carpeta que más vas a usar y modificar. Todo lo que crees lo crearás aquí dentro.
  • public/ — Archivos públicos
    Todo lo que pongas aquí estará disponible directamente en la web (por ejemplo, imágenes, favicon, etc.). Normalmente no necesitas tocarla mucho.

📄 Archivos importantes (explicación detallada)

🎯 Entender estos archivos es CLAVE para trabajar con React:

🏠 index.html (raíz del proyecto)

  • ¿Qué es? Es el "cascarón" estático de tu aplicación
  • ¿Qué contiene? Un HTML básico con un <div id="root"></div>
  • ¿Para qué sirve? Es donde React se "monta" y toma control
  • ¿Lo modificas? 🔸 Rara vez - Solo para favicon, meta tags o fuentes
  • 🚨 Importante: Aquí NO pones tu UI de React. Es solo el "contenedor"

⚡ src/main.jsx (punto de entrada)

  • ¿Qué es? El "encendedor" de tu aplicación React
  • ¿Qué hace? Arranca React y monta el componente raíz
  • ¿Lo modificas? 🔹 Muy poco - Solo para configuraciones globales
  • 📝 En nuestro curso: Solo lo tocamos para importar CSS global

🧭 src/App.jsx (configuración de rutas)

  • ¿Qué es? Donde definimos todas las rutas de nuestra app
  • ¿Qué hace? Configura React Router y conecta URLs con páginas
  • ¿Lo modificas? ✅ Aquí trabajamos - Cada vez que añadimos páginas
  • 📝 En nuestro curso: Contiene BrowserRouter, Routes y Route

🏗️ src/components/Layout.jsx (estructura común)

  • ¿Qué es? Define el diseño común (header, footer, navegación)
  • ¿Qué hace? Envuelve todas las páginas con la misma estructura
  • ¿Lo modificas? ✅ Aquí también - Para cambiar diseño global
  • 📝 En nuestro curso: Contiene header con menú + Outlet + footer más menu lateral si queremos. Es la plantilla donde entrará nuestro contenido.

🧭 En resumen para ti como alumno

Archivo Función principal ¿Se toca mucho?
index.html Contiene el <div id="root"> donde React se "inyecta" 🔸 Rara vez
main.jsx Arranca React y monta el componente raíz 🔹 Muy poco
App.jsx Contiene la configuración de rutas de tu app ✅ Aquí trabajamos
Layout.jsx Define el diseño común (header, footer, etc.) ✅ Aquí también

🎯 Estrategia de trabajo en nuestro curso:

  • 📁 Layout.jsx → Crear una estructura con header, menú y footer
  • 🧭 App.jsx → Definir rutas para conectar URLs con páginas
  • 🔧 main.jsx → Intentar tocar lo menos posible
  • 🏠 index.html → No lo tocamos

📄 ¿Cuál será nuestra página de inicio?

Crearemos src/pages/Home.jsx que será nuestra página principal:

  • src/pages/Home.jsx → Esta será tu página de bienvenida
  • Componentes de presentación: Elementos visuales que muestran información relevante.
  • Hero section: Parte superior con título grande, imagen o fondo llamativo y mensaje principal.
  • Features destacadas: Sección con las características principales, ventajas o beneficios, normalmente en tarjetas o lista.
  • Call to action: Invitación clara para que el usuario realice una acción (registrarse, descargar, contactar, etc.), suele ser un botón destacado.

💡 ¿Por qué no usamos App.jsx como inicio?

  • ✅ Separación clara: App.jsx solo para rutas
  • ✅ Escalabilidad: Fácil añadir más páginas
  • ✅ Mantenibilidad: Cada archivo una responsabilidad
  • ✅ Es una opción: Se usa mucho en proyectos reales especialmente si crecen.
  • ✅ Facilita el aprendizaje: Entiendes mejor la estructura de React

🏗️ Estructura recomendada al principio

Para que aprendas de forma organizada, te recomiendo esta estructura:

📁 tu-proyecto/
├── 📄 index.html              ← El "cascarón" estático
├── 📁 src/
│   ├── 📁 components/         ← Componentes reutilizables
│   │   ├── 📄 Header.jsx      ← Cabecera con navegación
│   │   ├── 📄 Footer.jsx      ← Pie de página
│   │   └── 📄 Tarjeta.jsx     ← Componente reutilizable
│   ├── 📁 pages/              ← Páginas de tu aplicación
│   │   ├── 📄 Home.jsx        ← Nuestra página principal (inicio)
│   │   ├── 📄 Proyectos.jsx   ← Página de proyectos
│   │   └── 📄 Contacto.jsx    ← Página de contacto
│   ├── 📄 Layout.jsx          ← Header/Footer + <Outlet/>
│   ├── 📄 main.jsx            ← Router y montaje de React
│   └── 📄 App.jsx             ← Configuración de rutas
└── 📁 node_modules/           ← No tocar nunca

🎯 ¿Por qué esta estructura?

  • 📁 components/ → Todo lo reutilizable va aquí
  • 📁 pages/ → Cada "pantalla" de tu app tiene su archivo
  • 📄 Layout.jsx → Lo que se repite en todas las páginas (Header, Footer)
  • 📄 main.jsx → Configuraciones globales y rutas
  • 📄 App.jsx → Lo vamos a usar para las rutas.

🧠 Flujo de trabajo en React (paso a paso)

Entender el flujo te ayuda a debuggear y crear mejor:

  1. Usuario abre tu app → Se carga index.html
  2. index.html carga → src/main.jsx se ejecuta
  3. main.jsx dice: "Monta React en #root y usa estas rutas"
  4. React Router ve la URL → Decide qué página mostrar
  5. Se renderiza Layout.jsx → Header, Footer + la página específica
  6. La página usa componentes → Como Tarjeta, Button, etc.
  7. ¡El usuario ve tu app! → Todo funcionando y responsive

💡 Clave para principiantes: React es como un árbol. main.jsx es la raíz, Layout.jsx es el tronco, y tus páginas son las ramas con hojas (componentes).

📄 Archivos que puedes eliminar o ignorar

  • src/index.css y app.css — Archivo de estilos por defecto.
    Si no lo usas, puedes eliminarlo, pero recuerda eliminar también su importación en src/main.jsx para evitar errores al arrancar el proyecto. Puedes crear tus propios archivos CSS o usar librerías como MUI.
  • package.json — Lista de dependencias y scripts.
    No lo borres, pero no necesitas modificarlo al principio.
  • node_modules/ — Carpeta con todas las librerías instaladas.
    Nunca la modifiques ni la borres manualmente.
  • vite.config.js — Configuración avanzada de Vite.
    Puedes ignorarlo por ahora pero no lo borres.

📝 Archivos y carpetas que deberás crear

  • src/assets/ — Carpeta para imágenes, logos y otros archivos multimedia.
    Guarda aquí tus imágenes para usarlas en los componentes y páginas.
  • src/pages/ — Carpeta para tus páginas principales.
    Por ejemplo: Home.jsx, About.jsx, Contact.jsx. Así mantienes tu proyecto organizado.
  • src/components/ — Carpeta para componentes reutilizables.
    Ejemplo: Header.jsx, Footer.jsx, Card.jsx, Button.jsx.
  • src/styles/ — Opcional, para tus archivos CSS personalizados.
    Puedes crear styles.css o App.module.css si prefieres separar los estilos.

💡 Resumen teórico: Piensa que src/ es tu espacio de trabajo, public/ es el escaparate, y el resto son herramientas y configuraciones que hacen que todo funcione. Puedes eliminar src/index.css si no lo usas, y centrarte en src/App.jsx y src/main.jsx para empezar.

🎉 Paso 3 · Tu primer "¡Hola Mundo!" en React

Ahora viene la parte emocionante: vamos a hacer nuestro primer cambio y entender qué está pasando.

🔍 Abre y examina src/App.jsx

Verás algo parecido a esto:

function App() {
  return (
    <h1>Hola React 🚀</h1>
  )
}

export default App

🤯 ¡Momento "aja"! ¿Qué está pasando aquí?

  • function App() → Una función de JavaScript (como las que ya conoces)
  • return () → Devuelve algo (como return en una función normal)
  • <h1>...</h1> → ¡HTML dentro de JavaScript! Esto es JSX
  • export default App → "Oye, otros archivos pueden usar esta función"

✏️ Tu primera modificación (¡hazla ahora!)

Cambia el texto del h1 por algo personal:

function App() {
  return (
    <h1>¡Hola estoy aprendiendo React! 🎉</h1>
  )
}

export default App

Guarda el archivo (Ctrl+S) y mira tu navegador. ¡Cambió automáticamente!

🪄 ¿Por qué es esto mágico?

En HTML normal tendrías que:

  1. Cambiar el archivo
  2. Guardarlo
  3. Ir al navegador
  4. Actualizar la página (F5)

Con React + Vite: ¡Los pasos 3 y 4 son automáticos! El navegador se actualiza solo cuando guardas.

⚛️ ¿Qué es el DOM virtual?

React utiliza algo llamado DOM virtual (Virtual DOM). Es una copia ligera del DOM real que vive en memoria. Cuando haces un cambio en tu app, React compara el DOM virtual con el DOM real y solo actualiza lo que cambió, en vez de recargar toda la página. Esto hace que tu aplicación sea mucho más rápida y eficiente.

  • 🔍 React detecta los cambios en el DOM virtual.
  • ⚡ Actualiza solo lo necesario en el navegador.
  • 🚀 Evita recargar toda la página, mejorando el rendimiento.

Piensa en el DOM virtual como un borrador que React usa para decidir qué partes de la página necesita cambiar realmente.

🔗 ¿Cómo se conecta el DOM virtual con el navegador que cambia solo?

Cuando guardas tu archivo en React, Vite detecta el cambio y recarga automáticamente la app en el navegador. Pero no recarga toda la página: React revisa el DOM virtual, compara con el DOM real y solo modifica lo que cambió. Así, el navegador muestra el nuevo contenido al instante, sin refrescar todo.

  • 🖥️ Vite observa los archivos y avisa a React cuando hay cambios.
  • ⚛️ React actualiza el DOM virtual y calcula los cambios mínimos.
  • 🌐 El navegador recibe solo los cambios necesarios y los muestra al usuario.

Por eso parece "mágico": el navegador cambia solo porque React y Vite trabajan juntos para actualizar solo lo necesario, de forma rápida y eficiente.

🎯 JSX: La sintaxis que mezcla HTML y JavaScript

🧠 ¿Qué es JSX y por qué es revolucionario?

JSX (JavaScript XML) es la sintaxis que permite escribir HTML dentro de JavaScript. Es como tener lo mejor de ambos mundos: la estructura de HTML y el poder de JavaScript juntos.

🔍 JSX vs HTML: Las diferencias que DEBES conocer

❌ HTML tradicional

<div class="mi-clase">
  <label for="nombre">Nombre:</label>
  <input type="text" id="nombre">
  <img src="foto.jpg" alt="Descripción">
</div>

✅ JSX (React)

<div className="mi-clase">
  <label htmlFor="nombre">Nombre:</label>
  <input type="text" id="nombre" />
  <img src="foto.jpg" alt="Descripción" />
</div>

⚡ Las 7 reglas de oro del JSX

1. 🏷️ className en lugar de class

❌ Incorrecto:

<div class="mi-estilo">Texto</div>

✅ Correcto:

<div className="mi-estilo">Texto</div>

¿Por qué? En JavaScript, class es una palabra reservada, así que JSX usa className.

2. 🔗 htmlFor en lugar de for

❌ Incorrecto:

<label for="email">Email:</label>

✅ Correcto:

<label htmlFor="email">Email:</label>

¿Por qué? Similar a class, for es una palabra reservada en JavaScript.

3. 🔒 Todas las etiquetas deben cerrarse

❌ Incorrecto:

<input type="text">
<img src="foto.jpg">
<br>

✅ Correcto:

<input type="text" />
<img src="foto.jpg" />
<br />

¿Por qué? JSX es más estricto que HTML. Toda etiqueta debe tener cierre.

4. 📦 Solo un elemento padre por componente

❌ Incorrecto:

return (
  <h1>Título</h1>
  <p>Párrafo</p>
);

✅ Correcto:

return (
  <div>
    <h1>Título</h1>
    <p>Párrafo</p>
  </div>
);

Alternativa: Usa <></> (Fragment) si no quieres un div extra.

5. 🧮 Expresiones JavaScript entre llaves { }

const nombre = "Ana";
const edad = 25;

return (
  <div>
    <h1>Hola {nombre}</h1>
    <p>Tienes {edad} años</p>
    <p>El próximo año tendrás {edad + 1}</p>
  </div>
);

💡 Clave: Todo lo que va entre { } se ejecuta como JavaScript.

6. 🎨 Estilos inline como objetos

❌ HTML tradicional:

<div style="color: red; font-size: 16px;">

✅ JSX:

<div style={{color: "red", fontSize: "16px"}}>

Nota: Propiedades CSS en camelCase (fontSize, backgroundColor, etc.)

7. 💬 Comentarios especiales

❌ No funciona:

<div>
  <!-- Comentario HTML -->
  <p>Texto</p>
</div>

✅ Correcto:

<div>
  {/* Comentario JSX */}
  <p>Texto</p>
</div>

Formato: {/* comentario */} para comentar dentro del JSX.

📝 Convenciones de nombrado: La base del código profesional

🎯 ¿Por qué importa tanto cómo nombres tus archivos?

Un buen nombrado no es vanidad: es comunicación. En 6 meses, cuando vuelvas a tu código, los nombres descriptivos serán la diferencia entre entender inmediatamente qué hace cada archivo o perder horas recordando tu propio trabajo.

🏷️ Reglas de oro para componentes React

✅ 1. PascalCase para componentes

❌ Mal nombrado:

  • header.jsx
  • button_component.jsx
  • my-card.jsx
  • UserProfile-component.jsx

✅ Bien nombrado:

  • Header.jsx
  • Button.jsx
  • Card.jsx
  • UserProfile.jsx

Regla: Primera letra mayúscula, cada palabra nueva con mayúscula, sin espacios ni guiones.

📂 2. Organización por carpetas lógicas

src/
├── 📁 components/           ← Componentes reutilizables
│   ├── 📄 Header.jsx       ← Cabecera del sitio
│   ├── 📄 Footer.jsx       ← Pie de página
│   ├── 📄 Button.jsx       ← Botón reutilizable
│   └── 📄 Card.jsx         ← Tarjeta de contenido
├── 📁 pages/               ← Páginas principales
│   ├── 📄 Home.jsx         ← Página de inicio
│   ├── 📄 About.jsx        ← Página "Acerca de"

├── 📁 hooks/               ← Hooks personalizados (avanzado)

    ├── 📁 images/
    └── 📁 icons/

Beneficio: Cualquier desarrollador encuentra lo que busca en segundos.

📝 3. Nombres descriptivos, no genéricos

❌ Nombres vagos:

  • Component.jsx → ¿Qué hace?
  • Item.jsx → ¿Qué tipo de item?
  • Thing.jsx → Muy genérico
  • Container.jsx → ¿Contiene qué?

✅ Nombres claros:

  • ProductCard.jsx → Tarjeta de producto
  • BlogPost.jsx → Artículo del blog
  • LoginForm.jsx → Formulario de login
  • ShoppingCart.jsx → Carrito de compras

Pregúntate: "Si alguien ve este nombre, ¿entiende inmediatamente qué hace?"

🔤 Convenciones adicionales importantes

Variables y funciones: camelCase

// ✅ Correcto
const userName = "Ana";
const userAge = 25;
const handleButtonClick = () => { /* ... */ };
const fetchUserData = async () => { /* ... */ };

// ❌ Incorrecto

Constantes: UPPER_SNAKE_CASE

// ✅ Para valores que nunca cambian
const API_BASE_URL = "https://api.miapp.com";
const MAX_USERS_PER_PAGE = 20;
const THEME_COLORS = {
  primary: "#2196f3",
  secondary: "#ff9800"
};

🏗️ Consejos de desarrollo: Evita el "código espagueti"

⚠️ El mayor enemigo del principiante: El anidamiento excesivo

"Código espagueti" es cuando tienes tantos niveles de anidamiento que pierdes el hilo. Como cajas dentro de cajas dentro de cajas... ¡hasta que no sabes dónde estás!

🚫 Ejemplo de lo que NO debes hacer

❌ Componente "monstruo" (no hagas esto)

function App() {
  return (
    <div>
      <header>
        <nav>
          <ul>
            <li>
              <a href="#">
                <div>
                  <span>
                    <button>
                      <div>
                        <span>Inicio</span>
                      </div>
                    </button>
                  </span>
                </div>
              </a>
            </li>
            {/* 50 líneas más de anidamiento... */}
          </ul>
        </nav>
      </header>
      <main>
        <section>
          <div>
            <div>
              <div>
                {/* Y aquí pierdes la cuenta... */}
              </div>
            </div>
          </div>
        </section>
      </main>
    </div>
  );
}

Problemas:

  • 🤯 Difícil de leer: No sabes dónde empieza o termina cada sección
  • 🐛 Difícil de debuggear: Los errores son complicados de encontrar
  • ♻️ No reutilizable: Todo está mezclado en un solo bloque
  • 👥 Trabajo en equipo imposible: Otros desarrolladores se perderán

✅ La solución: Divide y vencerás

✅ Componentes separados y limpios

1. Componente principal simple:

// App.jsx - Solo estructura principal
function App() {
  return (
    <div>
      <Header />
      <MainContent />
      <Footer />
    </div>
  );
}

2. Header separado:

// Header.jsx - Solo responsable de la cabecera
function Header() {
  return (
    <header>
      <Logo />
      <Navigation />
    </header>
  );
}

3. Navegación separada:

// Navigation.jsx - Solo se encarga del menú
function Navigation() {
  return (
    <nav>
      <NavLink to="/">Inicio</NavLink>
      <NavLink to="/about">Acerca de</NavLink>
      <NavLink to="/contact">Contacto</NavLink>
    </nav>
  );
}

Beneficios inmediatos:

  • 👀 Fácil de leer: Cada archivo tiene una responsabilidad clara
  • 🔧 Fácil de mantener: Cambias solo lo que necesitas
  • ♻️ Reutilizable: Puedes usar Header en otras páginas
  • 🤝 Trabajo en equipo: Cada persona puede trabajar en un componente

🤝 HTML tradicional vs React

📄 HTML tradicional:

<!DOCTYPE html>
<html>
<body>
  <h1>Hola Mundo</h1>
</body>
</html>

⚛️ React (JSX):

function App() {
  return (
    <h1>Hola Mundo</h1>
  )
}

💡 Las diferencias clave:

  • Estructura: En React, cada "página" es una función que devuelve HTML
  • Reutilización: Puedes usar la misma función muchas veces y crear componentes reutilizables para diferentes partes de tu app
  • Dinamismo: El HTML puede cambiar basado en datos (¡lo veremos pronto!)
  • Integración: HTML y JavaScript están juntos, no separados
  • Estado: Los componentes pueden tener "estado" que cambia con el tiempo
  • Props: Los componentes pueden recibir datos a través de "props"
  • Ciclo de vida: Los componentes tienen un ciclo de vida que puedes controlar
  • Eventos: Manejo de eventos (clicks, formularios) de forma más sencilla. No hay Eventos.
    • Virtual DOM: React usa un DOM virtual para actualizaciones eficientes
  • JSX: Sintaxis que mezcla HTML y JavaScript
  • Programacion declarativa: Enfocada en el qué, no en el cómo. Escribes lo que quieres que ocurra (por ejemplo, mostrar una lista), y React se encarga de cómo hacerlo internamente.

🧩 Bonus: ¿Cómo se construye una página real en React?

Ahora que tienes tu proyecto funcionando, probablemente te preguntas: "¿Cómo creo una página web completa con React?"

�️ La filosofía de React: "Componentes como piezas de LEGO"

En React, construyes páginas combinando componentes (piezas reutilizables). Imagina que cada parte de tu página es una pieza de LEGO:

  • 🧱 Header → La cabecera con navegación
  • 🧱 Sidebar → Barra lateral
  • 🧱 Card → Tarjetas de contenido
  • 🧱 Button → Botones reutilizables
  • 🧱 Footer → Pie de página
